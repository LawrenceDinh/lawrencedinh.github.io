The original page was not a failure

My original portfolio was functional, readable, responsive, and professionally presentable. It introduced me, described my background, displayed my skills and projects, and made my résumé and contact information easy to find.

It also gave me something more valuable than a polished mockup: a real system I could use and evaluate.

Before building that version, many design decisions were still abstract. I knew the content I wanted to present, but I did not yet know how much space each section needed, which information felt repetitive, what recruiters might see first, or how the page would feel after scrolling through the entire thing.

The first implementation answered those questions.

It established the basic content inventory:

  • professional identity;
  • career summary;
  • skills and tools;
  • work experience;
  • projects;
  • credentials;
  • contact information;
  • résumé and profile links.

It also established the original dark visual identity, responsive structure, and working interactions. Those foundations mattered because the redesign did not begin from an empty page. It began from a functioning portfolio with visible strengths and visible limitations.

The original page was not wasted effort that had to be replaced. It was effectively the prototype and research phase for the redesign.

What began to feel wrong

No single element of the first design was seriously broken. The weakness came from the way several reasonable choices accumulated.

Large rounded panels created clear separation, so I used them repeatedly. The introduction had a card. Contact methods had cards. Statistics had cards. Skills had cards. Employers and projects followed similar patterns.

The result was consistent, but consistency alone did not create meaning. Different kinds of information began to feel interchangeable.

An employer, a programming skill, a contact link, and a career statistic did not have the same role, yet the interface often presented them with similar visual weight. The page was organized, but it was not strongly differentiated.

The layout was also vertically expensive. Large containers, repeated padding, and card grids caused information to spread down the page. A visitor interested primarily in my projects or professional experience had to locate those sections through scrolling. The page determined the order in which the information would be discovered.

The hero carried too many responsibilities. It introduced me, summarized my background, presented action buttons, displayed career statistics, and sat beside a separate contact panel containing some of the same actions. The information was useful, but its placement was not always efficient.

Most importantly, the design did not strongly represent the kind of work I wanted the portfolio to communicate.

My background crosses software development, system testing, autonomous-vehicle validation, data analysis, robotics, technical operations, and motorsports. The old page described those areas, but its visual structure could have belonged to almost any software portfolio.

It looked competent. It did not yet feel specific.

Changing the navigation model

The largest improvement did not come from changing colors, borders, typography, or animation.

It came from changing the navigation model.

The original portfolio behaved like a document. Its information was arranged as a sequence of sections, and the primary method of exploration was scrolling.

The Enhanced version behaves more like an interface.

The new structure uses multiple navigation levels, each with a distinct responsibility.

The main navigation moves between major destinations and sections of the site:

  • About;
  • Featured;
  • Experience;
  • Skills;
  • Portfolio;
  • Projects;
  • Writing.

Inside the homepage, the Career Snapshot provides a second level:

  • Overview;
  • Experience;
  • Projects;
  • Credentials.

Below those levels, individual records and pages provide the actual detail:

  • employer contributions;
  • project case studies;
  • credential information;
  • complete writing articles;
  • dedicated project pages.

This creates a hierarchy:

Site destination → information category → detailed record

That hierarchy matters because a visitor no longer has to discover everything in the order imposed by the page.

Someone interested in projects can move directly toward project information. Someone evaluating my employment history can inspect Experience. Someone scanning quickly can remain in Overview and still see the major employers, featured projects, credentials, and contact actions.

The main navigation and the Career Snapshot tabs may look related, but they solve different problems. The main navigation answers, “Where am I going?” The sub-navigation answers, “What level of information do I want within this page?”

That distinction changed the portfolio from a long résumé presentation into a compact professional workspace.

Designing for scanning without becoming shallow

Recruiters and hiring managers may not explore a portfolio in the same way that its creator does. Some will inspect only the first screen. Others will open one project, confirm a résumé, or look for a contact link.

That made the Overview especially important.

The redesigned first desktop viewport can communicate:

  • my name and professional role;
  • a short positioning statement;
  • my primary capability domains;
  • three major employers;
  • four featured projects;
  • core credentials;
  • résumé, contact, and profile actions.

The old design used a similar amount of initial screen space for the introduction, statistics, contact information, and the beginning of the Skills section. The Enhanced design uses that space to present a broader professional map.

The goal was not to compress every subject into a vague label. Compression is useful only when the path to more detail remains clear.

The Overview therefore acts as an index rather than a replacement for deeper content. Employer cards summarize roles and domains. Project rows explain the purpose of each system. Credentials remain compact. “View Details” and “View Projects” links provide direct movement into focused information.

This is progressive disclosure: reveal enough to establish relevance, then make deeper material available when the visitor asks for it.

The structure is designed to support faster scanning, but it has not yet been proven through formal recruiter studies or analytics. Dense layouts can create their own problems. Small metadata, unfamiliar tab behavior, or subtle interaction cues may be obvious to me because I built the interface but less obvious to a first-time visitor.

The design is a stronger hypothesis about how the information should work. External use will determine where that hypothesis still needs adjustment.

Different content needs different interfaces

One of the most important lessons was that visual consistency does not require every type of content to use the same component.

The first design leaned heavily on reusable cards. That made implementation straightforward, but it also flattened distinctions between information types.

The Enhanced design keeps a shared visual system while giving different content different structures.

Professional experience uses employer records. Each record combines a logo, organization name, role, and compact responsibility summary. The focused Experience view can then prioritize chronology, domains, and contributions.

Projects use structured rows in the Overview because the immediate question is different: what was built, what technology was involved, and what capability does it demonstrate? On the dedicated Projects page, the layout can become more asymmetric because the projects have different levels of maturity and depth.

Credentials use a compact ledger. A certification does not need the same amount of space as a project or employer. Grouping credentials into a narrow record makes them visible without allowing them to dominate the page.

Contact information remains in a persistent side rail. It is operational information: email, résumé, profiles, location, and availability. Separating it from the central career narrative keeps the actions accessible without repeatedly inserting them into the main content.

I also learned where asymmetry did not work.

Making one NVIDIA experience card substantially larger than the other employers initially seemed like a way to emphasize my most relevant role.

In practice, it made the employment section feel like a project gallery and created awkward empty space. Chronology and contribution mattered more than visual dominance.

The same asymmetric approach works better for projects because one system can genuinely have greater scope, evidence, and maturity than another.

Density is not the same as clutter

The Enhanced version contains more useful information in less vertical space, but increasing density was not automatically an improvement.

Several iterations fit more content and still felt worse.

At times, I reduced text too aggressively to keep the Career Snapshot within a fixed desktop height. The information technically fit, but readability suffered. Other versions used too many borders, labels, or cyan accents. The sections were organized, yet the page felt visually active because everything competed for attention.

The difference between density and clutter came from hierarchy.

Density became manageable when the layout relied on:

  • consistent columns;
  • aligned baselines;
  • restrained borders;
  • predictable spacing;
  • clear heading levels;
  • monospaced metadata;
  • limited accent color;
  • repeated interaction patterns;
  • strong section boundaries.

The interface does not need a large empty gap around every item if alignment already explains the relationship between them.

This was a shift from using containers as the primary method of organization to using structure. Borders, rows, spacing, typography, and positioning now work together to show what belongs together.

There is still a tradeoff. The design was developed around a relatively large desktop viewport. A dense interface cannot simply be scaled down until it fits a phone. Mobile layouts need their own priorities, reading order, touch targets, and text sizing.

The desktop version can be deliberately compact. Responsive versions still need to remain comfortably readable.

Restraint improved the visual identity

The redesign draws influence from aerospace, defense, OSINT, and technical-system interfaces. That direction created another risk: making the portfolio look like fictional military software rather than a professional site.

Several visual ideas were reduced or rejected:

  • excessive cyan glow;
  • decorative HUD elements;
  • large background gradients;
  • animation on every component;
  • unnecessary system numbering;
  • labels treated as metadata when ordinary language was clearer;
  • angular controls repeated until they lost hierarchy;
  • effects that attracted more attention than the content.

The final direction uses near-black and dark navy surfaces, thin technical borders, restrained cyan accents, sharp geometry, and monospaced secondary text.

Cyan is most useful when it communicates state, hierarchy, or action. When every border and heading is bright cyan, the accent stops providing information.

The same principle applies to animation. A name-scramble effect, subtle panel response, or state transition can add personality. Animating every label would make the page feel unstable.

The interface became more distinctive as it became more restrained.

The goal was not to create a simulated classified terminal. It was to borrow the clarity and density of technical systems while keeping the content credible and readable.

AI accelerated implementation, not judgment

ChatGPT and coding agents changed the speed of the redesign substantially.

They reduced the time required to prototype layouts, revise CSS, compare interaction models, rewrite labels, inspect regressions, and apply changes across related pages.

That speed made experimentation inexpensive.

I could identify a problem in a screenshot, describe the intended correction, review an implementation, and repeat the process within the same session. Ideas that might previously have remained sketches could become functioning versions quickly enough to evaluate directly.

But implementation speed did not eliminate design judgment.

AI could produce a larger Experience card. It could not decide whether emphasizing one employer that way improved the information hierarchy.

It could make text smaller until a panel fit. It could not determine whether the result remained comfortable to read.

It could add another animation, another tab, another border, or another content module. It could not decide whether the added feature improved recruiter understanding or merely increased complexity.

The human work remained:

  • defining the actual problem;
  • determining what information mattered;
  • evaluating screenshots;
  • identifying why a version felt wrong;
  • rejecting functional but weak solutions;
  • preserving behavior across iterations;
  • locating conflicting CSS rules;
  • deciding which layout suited each content type;
  • recognizing when the design needed restraint.

The original page and the Enhanced redesign may have taken similar amounts of calendar time. The second version looks substantially more labor-intensive because AI compressed implementation while the first version compressed discovery.

The old page had already taught me what content existed and which problems mattered. AI then made it possible to test responses to those problems much faster.

That speed also created a new risk: over-iteration. When every idea is inexpensive to implement, it becomes easy to continue changing a system after it is already effective.

The ability to build more does not automatically justify building more.

Rejected approaches were part of the design

Many intermediate versions worked technically but did not support the final direction.

I tried or considered:

  • using large rounded cards for nearly everything;
  • keeping every capability inside the hero;
  • making the NVIDIA experience record much larger than the others;
  • repeating Overview content almost unchanged in detailed tabs;
  • using cyan across too many borders and surfaces;
  • adding interactions before the section hierarchy was stable;
  • preserving a visible mode toggle before both modes were equally supported;
  • using asymmetry in sections where chronology mattered more;
  • fitting panels by continually shrinking typography.

Those attempts were not necessarily wasted.

A rejected layout can reveal that two pieces of information should not have equal weight. A crowded panel can expose an unnecessary content block. A broken hierarchy can show that styling began before the role of the section was clear.

The final interface looks relatively simple: panels, rows, tabs, borders, text, and a contact rail. That simplicity conceals the number of decisions required to determine where each element belonged.

What the redesign still does not prove

A more refined visual system does not automatically prove good UX.

The Enhanced portfolio still needs validation beyond my own use of it.

Useful tests include:

  • whether a cold reader understands my professional role quickly;
  • whether visitors distinguish main navigation from page-level tabs;
  • whether keyboard users can operate every interaction;
  • whether screen-reader structure reflects the visible hierarchy;
  • whether small labels remain legible across devices;
  • whether mobile layouts preserve the intended reading order;
  • whether users discover deeper Experience and Project views;
  • whether the site loads quickly with additional screenshots and writing;
  • whether recruiters find the information they actually need;
  • whether analytics show that visitors engage with the deeper content.

The legacy Simple mode is not currently my development priority. The Enhanced version is the active product. Until the older mode is intentionally restored, it should not be treated as an equally maintained alternative.

That limitation does not invalidate the redesign. It separates a strong design hypothesis from validated behavior.

The current version provides a clearer architecture and more deliberate experience. Testing will reveal where the implementation still differs from the intention.

The page became a system

The most meaningful result of the redesign is not the visual theme. It is that the portfolio now behaves like a system capable of accepting new material.

The structure can support:

  • additional project case studies;
  • deeper experience records;
  • new credentials;
  • technical screenshots;
  • architecture diagrams;
  • project retrospectives;
  • research notes;
  • writing across software, robotics, data, and motorsports;
  • future project categories without expanding the homepage indefinitely.

The first design presented my information.

The Enhanced design organizes that information around how someone might want to explore it.

The same system could expand beyond the homepage. The dedicated Projects page retains the shared typography, borders, navigation, and metadata while giving project scope and maturity more room to shape the layout.

The Writing archive extends the same system in a different direction. Instead of employer records or project hierarchy, it uses editorial headings, publication metadata, article categories, and reading paths.

The next stage should therefore focus less on another complete visual reinvention and more on stronger evidence: working demonstrations, project architecture, screenshots, measured outcomes, technical decisions, and writing grounded in actual experience.

The most important lesson was that refinement did not come from adding more visual effects. It came from assigning every piece of information a clearer place, purpose, and path.
